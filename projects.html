Dynamic Customisable Reports
for Orange Umbrella

Orange Umbrella, a workplace culture consultancy, wanted to increase the use of the custom report builder. Using data from customer support, as well as analysis of use patterns, we developed a set of improvements that would make the tool more approachable for users.

Research

Methodology

With the goal of the project defined around increacing the use of the reports tool by existing customers, 3 primary sources of research were defined; Existing support documents, interviews with current users, and existing useage data. We decided against scenario testing after consultation with the client, as we agreed that it was unlikely to reveal any insights not available in the existing useage data.

Current UX

The current UX flow in the reports is common to many CRUD type applications; first a report is created, then edited until it is in the state that the user wants. The process of creating and developing a report can be represented in a flow diagram.

It is possible to define every report while staying within this flow, so there is no functionality missing from the system as is.

There are 3 sets of data that can be used to identify points in this flow which are associated with positive and negative feedback from users. Usage Pattern Analysis

In addition to user support queries, the format of the existing reports can be grouped into just a few types. Importantly, these are identicle to each other in structure, except from the root node (always an organisation), and in some cases the number of sub nodes.

The analysis of existing reports shows two groups of reports:

  1. Reports with one data grouping:
    • Responses for each round for a given oranisation
    • Responses for each phase for a given organisation
    • Responses for each Sub-Organisation for a given oranisation
    • Responses for each Survey for a given oranisation
    • Responses for each Attribute value, for a given attribute, for a given oranisation
  2. Reports with more than one data grouping—these are complex or compound reports like:
    • Responses for each Attribute value for a given attribute, for each round, for a given oranisation
    • Responses for each Attribute value for a given attribute, for each phase, for a given oranisation
    • Responses for each survey, for each round, for a given oranisation
    • Responses for each survey, for each phase, for a given oranisation
    • Responses for each sub-organisation, for each round, for a given oranisation
    • Responses for each sub-organisation, for each phase, for a given oranisation

This looks daunting, but when viewed as a tree, it's clear that there really is only a few different types of report, and they can be created programatically:

The complex reports can also be represented as a tree:

Going back to the existing saved reports in the system, we can now group them into 4 catgories:

  • Simple–the report generated fits one of the simple reports
  • Modified Simple–the report generated fits one of the simple reports, with some changes after generation
  • Complex –the report generated fits one of the complex reports
  • Modified Complex–the report generated fits one of the complex reports, with some changes after generation

For 60% of existing reports, the patterns matched above are all the report authors will need.

For the remaining 40%, the differences between their setup, and the nearest template can be cataloged and grouped. The transformations needed to get from a 'common report' to the final report can be cataloged and tallied.

The current operations possible on the node tree are Delete and Add child node.

The analysis showed that adding 3 operations would make all of the transformations possible within a few steps:

  • Adding a node above
  • Merging Nodes with the same parent
  • Changing the parent of a node

Concept Development

Existing

As this is a plan to improve an existing code base. The architecture of the current build acts as a useful constraint. It consists of 3 parts:

Bringing in the developers

At this point I scheduled a meeting with the developers to introduce them to the problems discovered in the research, and to discuss the feasibility of fixes.

The set of changes were broken up into 3 parts, and during a two hour whiteboard session we sketched out some preliminary designs for each. The sections were:

  • Report Template Sketch

    After looking at the research, decided that we could create a template based system for creating new reports. We would create template options dynamically, depending on what kind of surveys the users' organisation had. So an organisation without sub-organisations would never be presented with the option to create a sub-organisation report.

  • Report Body Sketch

    The reports themselves function well. However, there is concern that with increaced useage, the slow loading time of the report body could become a UX issue. To head this off, we decided to redesign the reports in a paginated form.

  • Node Editor

    The current node editor has significant technical debt, and some refactoring work needed to be done before any changes can be done. In addition, complex user interactions are difficult to scope–what seems simple can often be very time consuming to implement across browsers. Instead of agreeing on a drawing, we agreed on a precedence for the operations in order of risk; Add node below, Add node above, Merge nodes, and finally, Change parent node.

Developed Design

Template chooser

Wireframes

Once that a template based strategy has been decided on, it's time to decide how to present these options to users. 3 different layouts are developed and quickly tested:

  • The Carousel is a simple and effective way to scroll through many options, and there can be as many as 15 or more different template options generated.
  • Side navigation allows for more reports to be visible, and additional reports can be scrolled down to. There is also an additional affordance in the report preview.
  • The simplest layout, this displays the available templates in a flat hierarchy, using a grid layout. It is the simplest to build, but more importantly, it's the simplest to navigate. Users only need to be aware of one scrolling 'area' and as new templates become available, they are inserted into the grid. We decided to continue with this layout.

    • Visual Design

    At this point a few more affordances are added to complete the page. A link to saved reports, a chooser for the root node for templates for organisations with many sub organisations. We also bring forward the split between Simple and Complex reports that was useful in the research stage.

    In order to help the users understand what kind of results are in the report, we show a chart of response numbers. This illustrates the number of data series the report will have, and acts as a heuristic for users—they often know the response numbers for each group. It's also a relatively computationally light way to differentiate the graphs.

    In order to de-clutter the graphs, the series are not labeled. In order to see the series label, you have to hover over. Clicking a chart sends you to the Report body.
